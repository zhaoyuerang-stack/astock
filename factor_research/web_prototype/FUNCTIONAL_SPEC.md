# A 股策略工厂控制台功能文档

## 1. 产品定位

这是一个本地静态 Web 原型，用来把当前 A 股策略系统从脚本、JSON、日志和 Markdown 文档，整理成一个更接近实盘使用方式的控制台。

第一版目标不是做生产交易系统，也不是营销展示页，而是回答日常操作中最关键的问题：

- 今天系统建议持仓还是空仓？
- 当前启用的是哪一个策略版本？
- 推荐买入、卖出、观察哪些股票？
- 策略历史收益、回撤和目标差距如何？
- 数据是否新鲜，信号是否可信？
- 策略工厂有没有产生可进入台账的新母策略？
- 定时更新、日志和 stale gate 是否正常？

当前版本是纯静态 HTML，不接后端，不启动服务，不写生产数据。页面数据来自 `assets/mock-data.js` 中的静态快照，后续可以由脚本从真实 JSON 报告自动导出。

## 2. 用户与使用场景

主要用户是策略系统 owner，也就是每天需要看信号、看策略表现、看数据状态、决定是否继续推进研究的人。

典型使用流程：

1. 盘后打开 `index.html`。
2. 先看数据是否达到应有交易日。
3. 如果数据新鲜，再看今日信号、买入、卖出、持仓和观察名单。
4. 如果数据不新鲜，进入 `ops.html` 查看 daily update 报告和日志入口。
5. 如果想看策略生产进展，进入 `factory.html` 查看岛屿搜索、孵化池和失败原因。
6. 如果想确认哪些策略可以正式使用，进入 `registry.html` 查看在册、候选、孵化和退役状态。

## 3. 信息架构

原型包含四个页面：

| 页面 | 文件 | 主要问题 |
| --- | --- | --- |
| 实盘总览 | `index.html` | 今天怎么操作，当前策略收益如何，数据是否可信 |
| 策略工厂 | `factory.html` | 自进化搜索产生了什么候选，为什么还没过 precheck |
| 运维调度 | `ops.html` | 定时更新是否正常，数据是否 stale，日志在哪里 |
| 策略台账 | `registry.html` | 哪些策略在册、候选、孵化或退役 |

公共资源：

| 文件 | 用途 |
| --- | --- |
| `assets/styles.css` | 页面布局、表格、状态徽标、响应式样式 |
| `assets/app.js` | 页面渲染逻辑 |
| `assets/mock-data.js` | 静态快照数据 |

## 4. 实盘总览功能

`index.html` 是第一屏，也是最接近用户日常使用的页面。它不强调系统内部算法，而是优先展示可执行信息。

### 4.1 顶部状态

顶部状态栏展示：

- 系统名称。
- 当前阶段。
- 最新数据交易日。
- 应有交易日。
- 数据是否新鲜。
- 当前信号日期。

判断规则：

- `dataFreshness.latestTradeDate >= dataFreshness.expectedTradeDate` 时，数据视为新鲜。
- 数据不新鲜时，页面必须明确提示，不应把旧信号包装成新建议。

### 4.2 今日建议

今日建议卡片展示当前最重要的操作结论：

- 当前择时：持仓或空仓。
- 最新动作：例如 `空仓观望`、`买入`、`卖出`、`调仓`。
- 下一步决策：例如等待数据补齐、执行信号、检查日志。

当前快照状态：

- 最新生产信号日期是 `2026-06-01`。
- 应有交易日是 `2026-06-03`。
- 数据不新鲜。
- 当前信号为 `空仓观望`。
- 因为数据 stale，页面不应生成新的买入建议。

### 4.3 当前仓位

当前仓位卡片展示：

- 账户模式，例如模拟实盘。
- 现金比例。
- 股票持仓比例。
- 计划持仓数。
- 策略杠杆。

当前快照状态：

- 账户模式为 `模拟实盘`。
- 现金比例为 `100%`。
- 股票持仓比例为 `0%`。
- 持仓数为 `0`。

### 4.4 交易指令

交易指令区分三类列表：

- 买入。
- 卖出。
- 观察。

当前实现规则：

- 当 `signal.inMarket=false` 时，买入列表应为空或显示无买入。
- 当数据不新鲜时，不应展示新的强交易建议。
- 观察名单可以作为产品形态预览，但必须与真实买入建议区分。

当前快照中：

- 买入：无。
- 卖出：无。
- 观察：展示候选股票池预览，用来说明未来页面形态。

### 4.5 策略收益

策略收益区展示当前启用策略的核心表现：

- 年化收益。
- 最大回撤。
- Sharpe。
- 胜率。
- 目标年化和目标回撤差距。
- 权益曲线预览。

当前在册策略：

- 策略：`small-cap-size`。
- 版本：`v2.0`。
- 年化：`21.25%`。
- 最大回撤：`-16.21%`。
- Sharpe：`1.22`。

项目目标：

- 年化目标：`35%`。
- 最大回撤目标：`-15%`。

因此当前策略仍未达到最终目标，尤其是年化不足、回撤略超目标线。

### 4.6 数据状态

数据状态卡片展示：

- 最新交易日。
- 应有交易日。
- 数据是否新鲜。
- 抽样质量是否通过。
- 信号是否生成。

核心规则：

- 数据 stale 时，信号生成应该被 stale gate 拦截。
- 更新失败但数据已经是最新交易日时，可以继续生成信号，并在报告里标记。
- 更新失败且数据落后应有交易日时，不生成新信号。

## 5. 策略工厂功能

`factory.html` 用来展示自进化系统的研究和筛选结果。它面向策略研发，不直接给交易动作。

### 5.1 核心指标

页面展示：

- 评估策略数量。
- 进入 review 的候选数量。
- 通过 `registry_precheck` 的数量。
- 进入孵化池的数量。
- Pareto 前沿数量。
- 阶段验收状态。

当前快照：

- `evaluated=78`。
- `reviewCandidates=24`。
- `registryPrecheck=0`。
- `incubate=23`。
- `pareto=0`。
- `acceptance=false`。

结论：

- 阶段 1 还没有完成验收。
- 当前没有新的候选母策略可以进入正式台账。
- fundamental 方向目前更适合作为组合分散件，而不是独立母策略。

### 5.2 岛屿模型

页面展示不同岛屿的搜索状态，例如：

- fundamental。
- defensive。
- quality/value。
- regime。

每个岛屿展示：

- 运行状态。
- 候选数量。
- 收益。
- 回撤。
- 与在册策略相关性。
- 主要失败原因。

产品含义：

- 岛屿之间不迁移因子基因，只共享方法。
- 目标是找到逻辑不同、收益低相关、能通过 precheck 的母策略。

### 5.3 孵化池

孵化池展示还不够格进入台账，但值得继续观察的策略。

字段包括：

- 策略名称。
- 因子族。
- 年化收益。
- 最大回撤。
- 样本外收益。
- 压力回撤。
- 成本上浮后的表现。
- 与在册策略的相关性。
- 失败原因。

孵化池规则：

- 孵化不等于在册。
- 孵化候选不能直接用于生产信号。
- 孵化候选需要继续做降杠杆、降频、组合贡献测试，才可能进入 `registry_precheck`。

## 6. 运维调度功能

`ops.html` 展示每日数据更新和信号生成的运行状态。

### 6.1 Daily Update

页面展示最近一次 daily update 报告：

- 运行状态。
- 是否 dry run。
- 数据是否新鲜。
- 是否生成信号。
- 最新交易日。
- 应有交易日。
- 失败原因。

当前策略：

- 每个 A 股交易日盘后运行一次。
- 本机使用 macOS `launchd` 调度。
- 脚本不调用 OpenAI API，因此不会由 daily update 本身触发 429。

### 6.2 Launchd 任务

页面展示两个任务：

- daily update：工作日触发，脚本内部按中国盘后时间 gate。
- weekly maintenance：周末执行周频维护和完整质量校验。

### 6.3 只读命令

页面提供只读运维命令提示，用于：

- 查看日志。
- 手动触发任务。
- 查看 launchctl 状态。

这些命令只作为操作提示显示，静态页面不会执行命令。

## 7. 策略台账功能

`registry.html` 展示策略生命周期状态。

### 7.1 Active

Active 表示当前可以用于生产信号的策略。

当前在册策略：

- `small-cap-size / v2.0`。

该策略是当前唯一正式生产策略。

### 7.2 Candidate

Candidate 表示通过 `registry_precheck`，但还未正式启用的候选母策略。

当前状态：

- 候选批为空。
- `candidate_batch=[]`。

### 7.3 Incubate

Incubate 表示有潜力但还没有达到入册条件的策略。

当前状态：

- 孵化池数量为 `23`。
- 主要失败原因包括收益不足、压力回撤过大、相关性偏高。

### 7.4 Retired

Retired 表示已经退役或不再推荐使用的策略版本。

页面可以展示历史版本、退役原因和替代策略。

## 8. 数据结构

当前静态数据挂载在：

```js
window.ASTCOK_DATA
```

主要字段：

| 字段 | 含义 |
| --- | --- |
| `meta` | 系统名称、快照日期、阶段、项目目标 |
| `dataFreshness` | 最新交易日、应有交易日、数据新鲜度 |
| `signal` | 当前生产信号、择时状态、动作、持仓数量 |
| `liveTrading` | 模拟实盘视图、买卖计划、观察名单、收益曲线 |
| `ops` | daily update 报告、launchd 任务、运维命令 |
| `registry` | 在册策略、候选批、参考策略版本 |
| `factory` | 岛屿搜索、孵化池、precheck 和失败原因 |
| `roadmap` | 当前阶段与模块状态 |

未来真实数据来源：

| 来源 | 用途 |
| --- | --- |
| `signals/state.json` | 当前信号状态 |
| `signals/YYYY-MM-DD.json` | 每日生产信号 |
| `reports/ops/daily_update/*.json` | 每日更新报告 |
| `reports/islands_fundamental_1_12_parallel/summary.json` | 岛屿搜索汇总 |
| `reports/islands_fundamental_1_12_parallel/incubation_pool.json` | 孵化池 |
| `strategy_versions.json` | 策略台账 |
| `STATUS.md` | 阶段状态摘要 |

## 9. 核心业务规则

### 9.1 数据新鲜度优先

交易信号必须先通过数据新鲜度检查。

- 数据新鲜：可以展示当日信号。
- 数据 stale：不生成新的买入卖出建议。
- 更新失败但数据仍新鲜：允许生成信号，并记录异常。
- 更新失败且数据落后：禁止生成新信号。

### 9.2 在册策略才可生产

只有 Active 或明确切换为生产状态的策略可以生成生产信号。

- `incubate` 策略不能直接参与生产。
- `candidate` 策略需要通过验收后才能启用。
- `registry_precheck=0` 表示当前没有新增母策略可入册。

### 9.3 观察名单不是买入建议

观察名单用于提示系统关注方向，不等同于交易指令。

页面需要明确区分：

- 买入：可执行交易建议。
- 卖出：可执行交易建议。
- 观察：非执行信号。

### 9.4 收益必须采用真实成本口径

策略表现展示应优先使用包含交易成本、滑点和换手约束后的结果。

如果展示无成本结果，必须明确标注，不应与生产口径混用。

## 10. 后续迭代

建议按以下顺序推进：

1. 增加静态快照导出脚本，把真实 JSON 自动转换成 `assets/mock-data.js`。
2. 在 `signal.inMarket=true` 时展示真实买入、卖出、调仓明细。
3. 增加持仓明细，包括股票代码、名称、行业、权重、入选原因、风险标签。
4. 使用真实权益曲线替换当前静态曲线。
5. 增加单日信号归档查看。
6. 增加策略对比页面，比较 active、candidate、incubate 的收益、回撤、相关性和组合贡献。
7. 增加只读本地 API 或自动生成 HTML 报告，减少手工维护 mock 数据。

## 11. 验收标准

当前文档对应的原型应满足：

- 四个页面可以直接通过本地文件打开。
- 页面导航可用。
- 页面显示的关键数字与 `assets/mock-data.js` 一致。
- 用户能在首页直接判断今天是否有可执行交易动作。
- 数据 stale 时不会误导用户执行旧信号。
- 策略工厂页面明确说明当前没有通过 `registry_precheck` 的新母策略。
- 运维页面能定位 daily update、stale gate 和日志入口。
- 策略台账页面能区分 active、candidate、incubate、retired。
- 原型不修改回测、工厂、调度或生产数据逻辑。
