# Roadmap.md

# AI-Native 个人量化研究分析平台 Roadmap

版本：v0.1  
文档类型：项目路线图 / Product & Engineering Roadmap  
目标形态：产品级 Web 平台，而非 Streamlit 原型  
核心范式：左侧导航栏 + 中央研究工作区 + 右侧常驻 AI Agent  
核心定位：Research-first Quant OS / 个人投资研究审计与控制系统

---

## 1. Roadmap 总原则

本项目的建设目标不是做一个普通行情看板，也不是做一个自动荐股工具，而是做一个 AI-native 的个人量化研究平台。平台必须围绕以下闭环展开：

```text
假设提出 → 数据验证 → 因子研究 → 策略回测 → 组合管理 → 风险控制 → 实验沉淀 → AI 审查 → 报告输出
```

产品建设必须遵守 5 条原则：

1. 产品级架构优先，而不是一次性功能堆叠。
2. 研究可信度优先，而不是单纯追求收益曲线好看。
3. AI Agent 必须成本可控、可审计、可降级。
4. MVP 要能给真实用户使用，并能验证付费意愿。
5. 所有关键结果必须可复现、可追踪、可解释。

## 3. Phase 0：产品验证与可点击原型

### 3.1 阶段目标

用最小成本验证以下问题：

1. 用户是否理解“AI-native 量化研究工作台”的价值。
2. 三栏 Web 结构是否成立。
3. 右侧常驻 Agent 是否能形成明显差异化。
4. 用户是否愿意为该产品支付 ¥1,999–¥4,999 的内测费用。
5. 哪些页面最有感知价值：因子研究、策略回测、风险控制、AI 报告，还是实验管理。

### 3.2 核心交付物

```text
1. 产品级高保真 UI
2. 可点击原型
3. 9 个一级页面的静态展示
4. 右侧 AI Agent 交互样式
5. 产品介绍页 / Demo Script
6. 种子用户访谈问卷
7. Founding Pro 预售方案
```

### 3.3 页面范围

Phase 0 只需要静态或半动态页面：

```text
总览
数据中心
因子研究
策略回测
组合管理
风险控制
研究实验
AI研究助手
系统设置
```

### 3.4 验收标准

```text
1. 用户看完 5 分钟 Demo 后，能说清楚产品解决什么问题。
2. 至少 20 个目标用户完成访谈。
3. 至少 5 个用户表达明确付费意愿。
4. 至少 2 个用户愿意支付内测订金或加入 Founding Pro。
5. 明确 P0 页面优先级。
```

### 3.5 不做事项

```text
不接真实数据
不做真实回测
不做用户系统
不做支付
不做复杂后端
不承诺投资收益
```

```text
选择股票池 → 查看数据 → 运行基础因子 → 跑策略回测 → 生成 AI 报告 → 保存实验记录
```

### 4.4 AI Agent 初版能力

```text
1. 解释当前页面指标
2. 总结因子研究结果
3. 总结回测表现
4. 生成基础研究报告
5. 给出下一步实验建议
6. 提醒可能存在的过拟合风险
7. 解释基础风险预警
```

AI Agent 不允许做：

```text
自动下单
承诺收益
直接荐股
自动修改关键策略参数
删除实验
绕过额度限制
```

### 4.5 验收标准

```text
1. 用户可以登录并进入工作区。
2. 用户可以看到真实或半真实数据。
3. 用户可以完成至少 3 个基础因子研究。
4. 用户可以运行至少 2 个策略回测。
5. 每次回测可以保存 experiment_id。
6. 用户可以生成至少 1 份 AI 研究报告。
7. 右侧 AI Agent 能读取当前页面上下文。
8. 出错时有明确错误提示，不直接崩溃。
9. 所有 AI 调用经过 AI Gateway。
10. 所有 AI 调用记录到 ai_usage 表。
```

### 4.6 阶段成功指标

```text
试用用户：10–30 人
周活跃率：> 40%
完成研究闭环用户比例：> 30%
愿意继续使用用户比例：> 30%
明确付费意愿用户：> 5 人
严重数据/回测投诉：可解释且可修复
```

## 13. 风险与缓解策略

### 13.1 产品风险

| 风险 | 表现 | 缓解 |
|---|---|---|
| 功能过重 | 做了很多但用户不用 | 坚持 P0 闭环，先验证高频场景 |
| 变成普通看板 | 缺少差异化 | 强化 Agent + 实验审查 + 研究报告 |
| 非开发者用不懂 | 学习成本高 | 用流程化引导和 Agent 解释降低门槛 |
| 用户以为是荐股工具 | 合规和预期风险 | 明确定位为研究审计，不承诺收益 |

### 13.2 技术风险

| 风险 | 表现 | 缓解 |
|---|---|---|
| AI 成本失控 | 重度用户消耗过高 | AI Gateway + 点数 + 限额 |
| 回测错误 | 用户不信任 | 参数记录 + 数据版本 + 审查 Agent |
| 数据不稳定 | 结果波动 | 数据 QA + 可替换数据源 |
| 前端复杂度高 | 开发慢 | 组件库 + 页面模板 + AI 辅助开发 |

### 13.3 商业风险

| 风险 | 表现 | 缓解 |
|---|---|---|
| 用户不愿付高价 | 转化低 | Founding Pro 验证价格弹性 |
| 获客成本高 | CAC 过高 | 内容获客 + 社群 + 案例驱动 |
| 低价用户消耗资源 | 毛利低 | 年费高客单价 + 额度限制 |
| 机构版太早 | 项目拖慢 | 先个人 Pro，再 Team，再私有部署 |

## 16. 最终路线判断

本项目不应走“低价大众炒股工具”路线，而应走：

```text
高客单价
专业研究
AI-native
可复现实验
风险控制
个人到小团队逐步扩展
```

推荐主路线：

```text
Phase 0：验证产品概念
Phase 1：可试用内测
Phase 2：Founding Pro 收费验证
Phase 3：Pro 标准版规模化
Phase 4：Team 版提高客单价
Phase 5：私有部署打开机构收入
```

核心判断：

```text
不要先做机构级重系统。
不要先做低价大众版。
不要做自动荐股。
不要让 AI 成本失控。
先用产品级架构做收敛版 MVP，用真实付费验证价值。
```
