# MINIPROGRAM_ARCHITECTURE — 微信小程序架构设计

> 定位:**策略可行性审计 SaaS**——用户用自然语言描述策略,平台跑 9-Gate 审计,推送可行性裁决。
> 核心价值 = 让用户知道他的策略是否真的可行(还是过拟合/成本吞噬/样本外失效)。
> 信号、组合、行情是次要佐证,不是主角。9-Gate 记分卡是英雄页。
> 本文档先于代码,定边界、定合规、定节奏。
>
> **v2 修订(2026-07-03)**:核心价值从「展示今日信号+组合表现」pivot 到「策略可行性审计」。
> 首页从"今日研究"改为"审计入口"。合规口径从"研究参考"升级为"研究方法学工具",过审更稳。

| 维度 | 决策 | 影响 |
|---|---|---|
| **核心价值** | **让用户知道策略是否可行(9-Gate 审计)** | 首页=审计入口,信号/组合降为佐证 |
| 用户定位 | 公开上架获客 | 合规口径=研究方法学工具,非投资咨询 |

## 1. 整体架构

```
[微信小程序原生]
  ↓ wx.request (HTTPS, 域名白名单)
[FastAPI :8011]
  ├── /miniapp/v1/* (新增, 小程序专用)
  │     ├── 内容脱敏层 (信号→研究候选, 指令→观察池)
  │     ├── 数据聚合层 (一次请求拿首页所有数据)
  │     ├── openid 限流
  │     └── 免责声明注入
  └── /api/* (Web 端复用, 不动)
        ↓
[services/read/*] (只读复用, 不动)
  ↓
[data_lake / signals / reports / strategy_versions.json] (只读)
```

### 铁律不动层(任何小程序改动不得触碰)

遵守 AGENTS.md / CLAUDE.md 分层依赖与唯一写入口铁律:

- `core/engine.BacktestEngine` — 回测唯一权威
- `factory/*` — 策略工厂
- `production/*` — 生产层(run_daily / paper_engine / launchd ops)
- `strategy_registry.register()` — 台账唯一写入口
- `data_lake/` 写入口 — 数据湖只读消费

小程序只经 `services/read/*` 只读消费,绝不反向写入。

## 2. 信息架构(4 Tab + 2 子包)

### 主包(<1.5MB,首屏必需)

```
pages/
├── index/        首页 · 审计入口(输入策略 + 审计历史)
├── audit/        审计详情 · 9-Gate 记分卡(英雄页,在主包保证直达)
├── portfolio/    组合 · 表现(佐证:在册策略实盘跟踪)
├── me/           我的
components/        TDesign + 自定义图表
utils/            request / auth / subscribe
```

### 子包(按需加载,主包不膨胀)

```
subpackages/
├── audit-detail/    审计深页 · 逐门指标 + 净值曲线 + 分组收益 + 三段样本
├── strategy-detail/ 平台已审计策略库 · 9-Gate 热力图 · 血缘 · 衰减监控
└── portfolio-detail/ 净值大图 · 回撤对比 · 调仓历史
```

### 页面职责

**Tab1 首页「审计入口」** — 核心价值入口,首屏回答"你的策略可行吗"
- 顶部:大输入框「描述你的策略,9-Gate 审计可行性」+ 示例气泡
- 输入示例:"小市值低流动性月调仓,持仓 25 只"
- 提交 → spec 回显确认页("我理解你的策略是:股票池=全A / 调仓=月频 / 因子=小市值+低流动性,对吗?")
- 确认 → 入队,提示"审计约 5-20 分钟,完成后推送通知"
- 下方:我的审计历史(最近 5 条 + 状态:运行中/已完成/失败)
- 底部:固定免责声明条「研究方法学工具,非投资建议」

**Tab2 审计详情「9-Gate 记分卡」** — 英雄页,审计完成后用户盯着看
- 总体裁决大卡:VIABLE(绿) / FALSIFIED(红) / INCONCLUSIVE(黄)
- 策略摘要:用户原始描述 + 审计区间 + 样本外起点
- 关键指标 4 卡:年化 / 夏普 / 最大回撤 / DSR p-val
- 9-Gate 逐门裁决列表(Gate 0-8 + 7A),每门:状态点 + 名称 + 关键指标 + PASS/WARN/FAIL
- "为什么不可行/可行"解释卡(2-3 条 bullet,基于失败门)
- 「查看完整审计详情」→ 子包 audit-detail
- 底部免责条

**Tab3 组合「在册跟踪」**(原 Tab2,降级为佐证)
- 平台已审计通过(VIABLE)策略的实盘跟踪表现
- 净值曲线 + 持仓表 + 回撤
- 价值:证明"审计通过 ≠ 永远有效",展示持续监控

**Tab4 我的「我的」** — 身份与合规
- 微信登录卡 + 今日剩余免费审计次数
- 审计历史完整列表
- 订阅消息管理(审计完成/在册策略衰减告警)
- 研究使用声明(首次强制阅读 20s 倒计时 + 勾选确认)
- 完整免责声明

## 3. 审计闭环(新核心,异步 SaaS)

### 3.1 闭环流程

```
[小程序] 输入自然语言策略描述
   ↓
[后端] DeepSeek LLM 解析 → strategy spec JSON
   ↓
[小程序] 回显 spec,用户确认/修正
   ↓
[后端] 入队 audit_queue,返回 job_id + 预计等待
   ↓ (用户可离开)
[后端 worker] factory/ 构建 Signal → BacktestEngine canonical 回测 → NineGatesEvaluator 9 门
   ↓ (5-20 分钟)
[后端] 写 audit_history/{job_id}.json + 推送订阅消息
   ↓
[小程序] 用户点开 → 9-Gate 记分卡(英雄页)
```

### 3.2 spec schema(自然语言 → 结构化)

LLM 解析必须产出可执行 spec,字段:
- `universe`: 股票池(全A / 沪深300 / 中证500 / 创业板)
- `rebalance_freq`: 调仓频率(周/月/季)
- `holdings`: 持仓数量
- `factors`: 因子列表(从已实现因子库选:动量/价值/盈利/质量/波动率/流动性/小市值/反转)
- `weights`: 因子权重(等权/市值加权/自定义)
- `neutralization`: 中性化(无/市值/行业)
- `cost_model`: 成本模型(默认标准:手续费+滑点)

回显确认环节:用户可逐项修正,不直接跑。

### 3.3 审计结果裁决规则

| 裁决 | 条件 | 含义 |
|---|---|---|
| VIABLE(绿) | Gate 4 DSR p<0.05 且 Gate 6 成本通过 且 Gate 7 样本外不塌 | 统计上可信,可进入前向观察 |
| INCONCLUSIVE(黄) | 部分 WARN,DSR 边界 | 证据不足,建议前向纸面验证 |
| FALSIFIED(红) | Gate 4 FAIL 或 Gate 7 严重塌陷 | 无法排除过拟合/样本外失效 |

### 3.4 与现有系统的边界(铁律不动)

- 复用:`factory/` 构建 Signal、`BacktestEngine` 跑 canonical 回测、`NineGatesEvaluator` 跑 9 门
- **不碰 holdout 金库**:用户审计只走 IS(2018-2022)+OOS(2023+),不触碰 holdout boundary(2025+),金库只给平台正式晋级用
- **不进 strategy_registry**:用户审计结果写独立 `audit_history/`,不污染正式台账(那是 `register()` 唯一写入口)
- **不自动晋级**:VIABLE 只是"建议前向观察",不等于在册,在册仍需人工 `register()`

### 3.5 关键风险与对策

| 风险 | 对策 |
|---|---|
| 审计耗时 5-20 分钟 | 异步 + 进度查询 + 设预期文案,不让用户干等 |
| NL 解析可能出错 | spec 强制回显确认,不直接跑 |
| 算力成本(回测+LLM) | 每用户每日免费限额 3 次,超出排队/拒绝 |
| 并发瓶颈(单机) | MVP 单 worker 串行,job 队列 + 预计等待时间 |
| 9-Gate 为因子设计 | 用户"策略"多为组合:Gate 0-4 逐因子 + Gate 5-8 组合级 |
| holdout 金库保护 | 审计只走 IS+OOS,不碰 holdout,金库只给正式晋级 |
| 合规口径 | "研究方法学工具"非投资建议,免责仍需但风险降级 |

## 4. 合规工程化(公开上架 P0,过审关键)

### 3.1 内容脱敏层(services/read/miniapp.py 出口过滤)

| 原始语义(Web 端) | 小程序展示语义 |
|---|---|
| 买入/卖出指令 | 进入/移出研究观察池 |
| 调仓指令 | 研究组合调整参考 |
| 委托数量/价格/金额 | **不展示**,只展示比例和方向标签 |
| 具体股票代码 | 可展示,必须标注"研究候选,非投资建议" |
| 模拟盘交易记录 | 标注"模拟盘,非实盘"水印 |

### 3.2 强制免责(四道防线)

1. **每页底部固定条**:「本内容仅供研究学习,不构成任何投资建议」
2. **详情页顶部 banner**:可关闭但每次进入都弹
3. **分享卡片**:标题 + 缩略图 + 「研究参考」标签
4. **首次进入**:强制阅读《研究使用声明》20s 倒计时 + 勾选确认

### 3.3 无交易入口

- 不提供下单、不跳转券商、不做"一键跟单"
- 不展示"买入金额""委托数量"等交易语义字段
- 模拟盘数据全程"模拟盘,非实盘"水印

### 3.4 内容预审

- 关键词黑名单:必涨/稳赚/内幕/荐股/跟单/包赚/涨停板
- 所有展示内容经 `services/read/miniapp.py` 出口过滤
- 9-Gate 审计深页加"研究方法学说明"前置页(避免审核员不懂术语)

### 3.5 用户告知

- 首次启动:研究使用声明 + 风险揭示书
- 每次进入信号页:顶部「研究参考」条
- 分享出去的卡片:带免责标签

## 5. 推送机制(订阅消息,留存核心)

### 模板

| 模板 | 触发 | 内容 |
|---|---|---|
| 每日盘后研究简报 | 每日 17:00 | 市场状态 + 候选池摘要 + 组合收益 |
| 策略衰减告警 | DSR/decay 触发 | 策略名 + 衰减指标 + 建议 |
| 策略状态变更 | 台账 status 变更 | family/version + 旧→新状态 |

### 微信订阅消息机制要点

- 每次用户授权只能推一条,需做"循环授权"
- 每次用户查看详情页时,静默请求一次订阅授权
- 累积授权池,按优先级消费:衰减告警 > 状态变更 > 日报
- 日报推送给所有有授权的用户,告警只推送给相关策略关注者

## 6. 技术选型

### 前端:原生小程序(不上 Taro/uni-app)

理由:
- 公开上架审核,原生最顺,跨端框架有额外审核风险
- 性能最优,启动最快(双线程架构原生支持)
- 包体积最小,主包控制更容易
- 你们已有 Next.js Web,不需要跨端复用

### UI:TDesign Miniprogram + 自定义图表

- TDesign:60+ 组件,微信生态主流,审核友好
- 图表:手写 SVG/Canvas(净值曲线、回撤、热力图),不用 ECharts(包太大)

### 网络:复用 FastAPI:8011

- 新增 `/miniapp/v1/*` 路由前缀
- 统一鉴权(openid + session)
- 限流(openid 维度,防刷)

### 状态管理

- 轻量:MobX-miniprogram 或原生 globalData
- 不引入 Redux(过度设计)

## 9. 与现有 Web 的关系

- Web = 研究操作系统(深度工作台,AI Agent 协作,三栏高密度)
- 小程序 = 轻量输出端(日报 + 仪表盘 + 档案,单手即时)
- 共享后端 FastAPI:8011,路由分前缀(`/api/*` vs `/miniapp/v1/*`)
- 共享数据层 services/read/*,小程序多一层脱敏聚合
- 不共享前端代码(技术栈不同,场景不同)

小程序是 Web 的**输出投影**,不是 Web 的**功能子集**。
