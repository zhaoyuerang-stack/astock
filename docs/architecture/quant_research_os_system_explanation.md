## 1. 一句话定位

这套系统不是普通行情看板, 也不是单个策略回测脚本。它是一个面向个人/小型研究团队的 A 股全市场因子研究操作系统, 目标是持续完成:

```text
发现候选 alpha -> 证伪伪 alpha -> 登记可追溯策略 -> 监控衰减 -> 退役替换
```

系统的真正资产不是某一条策略, 而是:

- 全市场、日频、可审计的数据基础设施。
- 可批量生成候选的因子与策略工厂。
- 防未来函数、防过拟合、防幸存者偏差的确定性验证流程。
- 可追溯、可退役、可复测的策略生命周期台账。
- 面向研究决策而不是炫技展示的 Web 驾驶舱。

核心世界观: 任何单一策略默认会失效; 漂亮回测默认先假设为过拟合、数据污染、未来函数、成本低估或特定行情依赖。系统价值在于让研究循环持续、诚实、可复核。

## 2. 系统为什么存在

传统个人量化系统容易出现三个问题:

1. 回测入口太多: 每个脚本各算一套收益, 结果不可复现。
2. 研究结论太乐观: 搜了很多候选, 只展示幸存者, 没有诚实多重检验惩罚。
3. 展示层太像交易台: 页面充满收益曲线和绿灯, 但未回答"这个策略是否真的通过独立验证"。

本系统的第一性目标是把这些风险机械化地关住:

- 统一数据口径: 正式研究必须来自 `data_lake` 和 canonical 加载路径。
- 统一回测权威: 正式回测只能由 `core.engine.BacktestEngine` 产生。
- 统一晋级通道: 候选进入台账必须经过 `workflow` 和 `strategy_registry.register()`。
- 统一证据规则: 策略是否有效必须由确定性代码、统计检验、成本敏感性、9-Gate 和 holdout 规则决定, LLM 不做有效性裁判。

## 3. 静态架构: 七层系统

系统从底到顶可分成七层:

```text
data_lake
  -> lake / data loaders
  -> factors
  -> core.engine
  -> strategies + factory + workflow
  -> registry + portfolio + production
  -> services + FastAPI + Next.js
```

### 3.1 数据基础设施

数据层负责 A 股全市场、全历史、含退市股的日频研究口径。它包含价格、成交额、复权、不复权、财务、资金流、ETF、质量报告和数据指纹。

关键要求:

- 财务和公告数据必须按公告日/披露日对齐, 避免未来函数。
- 估值指标必须使用定义一致的价格口径, 不得用后复权价直接计算估值。
- 数据质量失败应 fail closed, 不允许静默换口径继续研究。
- 数据湖写入必须走受控路径, 并由守卫检查直写行为。

### 3.2 因子层

`factors/` 保存 canonical 因子公式、转换函数、AutoResearch DSL 运行适配和因子组件。因子层只负责计算信号, 不负责决定策略是否入册。

边界:

- 可以消费数据和 lake 工具。
- 不得依赖策略、工厂、workflow、services 或 API。
- alpha 因子和 policy filter 要区分; filter 只能作为宿主策略的排除规则, 不能包装成独立策略收益。

### 3.3 回测内核

`core.engine.BacktestEngine` 是正式回测的唯一权威。成本、T+1、shift 语义、融资成本、指标计算都在这一层统一。

这层解决的问题是: 不允许不同策略各自写一套看似合理但口径不同的回测代码。任何正式绩效都必须能由统一引擎复现。

### 3.4 策略与研究工厂

策略层负责已经明确的生产策略构造器。研究工厂负责生成候选、变异、交叉、AutoResearch、候选池、L0-L3 cheap-first 验证。

关键原则:

- 工厂可以提出大量候选, 但不能直接写正式台账。
- LLM 可以生成候选和抽取研报信息, 但候选去留由代码判断。
- 适应度函数必须把相关性、换手、成本、容量和多重检验纳入约束, 避免系统只复制已有小盘/非流动性暴露。

### 3.5 Workflow 晋级层

`workflow/` 是从研究候选到正式台账的唯一通道。典型路径:

```text
factory candidate
  -> phase1 synthetic anti-leakage audit
  -> phase2 backtest
  -> phase3 walk-forward
  -> nine_gate_runner
  -> phase4_register
  -> strategy_registry.register()
```

其中 `phase1_synthetic` 是机械执行防未来函数审计的关键闸门。`nine_gate_runner.py` 是可复用 9-Gate 入口, `scripts/research/run_nine_gates_all.py` 只是 CLI 包装, 不应被 workflow 反向依赖。

### 3.6 台账、组合与生产

`strategy_registry` 保存母策略 family 和版本 version。系统不是按孤立版本管理, 而是按母策略生命周期管理:

- 候选
- 观察/参考
- 在册
- 退役
- 已证伪

`portfolio/` 负责 runner、组合腿、deployment runner、paper engine 和优化辅助。生产层负责日更、信号、模拟盘、日志与运行产物, 但不得绕过台账和 readiness 规则。

### 3.7 服务层与 Web 决策驾驶舱

`services/` 是 API/Web 与研究系统内部之间的受控边界:

- `services/read`: 读模型和 artifact 查询。
- `services/actions`: 受控执行和 workflow 动作。
- `services/agent`: LLM 支持, 但不做 alpha 裁决。

FastAPI 路由应保持薄层, 通过 services 访问业务逻辑。Next.js 前端不是传统交易台, 而是沿 alpha 生命周期组织的决策驾驶舱。

## 4. 动态流程: alpha 生命周期

系统的核心闭环:

```text
假设 -> 候选生成 -> L0 -> L1 -> L2/L3 -> 9-Gate -> 登记 -> LIVE/观察 -> 监控 -> 退役
```

每一步的含义:

- 假设: 人或强模型提出机制驱动想法, 不能只靠收益曲线反推叙事。
- 候选生成: DeepSeek/API 模型或规则 DSL 生成大量候选。
- L0: 低成本 IC 扫描, 看是否有基础预测力。
- L1: 快速回测, 估计交易成本后的初步可交易性。
- L2/L3: regime、walk-forward、成本敏感、样本外压力。
- 9-Gate: 数据、假设、IC、风格中性、DSR/PBO、回测、容量、OOS、监控计划。
- 登记: 只经 `phase4_register` 和 registry 唯一入口。
- 监控: 衰减、漂移、风险超限、失效信号。
- 退役: 保留历史和证据, 不覆盖、不删除、不洗白。

## 5. 防自欺机制

系统最重要的设计不是"能自动跑", 而是"自动跑时不自欺"。

### 5.1 LLM 分工铁律

LLM 可以做:

- 候选因子生成。
- 批量 NLP 抽取。
- 研究假设草拟。
- 报告解释。
- 错误定位辅助。

LLM 不能做:

- 判断 alpha 是否有效。
- 判断是否入册。
- 替代回测、DSR、多重检验、成本敏感性或 holdout。
- 为了达标建议改样本、成本、shift、T+1 或股票池。

一句话: LLM 是燃料, 不是裁判。

### 5.2 Trial 账本

系统必须记录历史累计试过多少候选, 因为搜索越多, 漂亮结果越可能只是幸存者偏差。`governance/trial_ledger.py` 负责把搜索自由度计入 DSR 等多重检验惩罚。

### 5.3 Holdout 金库

holdout 是从未用于搜索的数据段, 只能在晋级前触碰。搜索、phase2/phase3、9-Gate 都必须截断到 boundary 之前。boundary 只能向前迁移, 且有 append-only 历史账本。

### 5.4 9-Gate 自证铁律

候选策略的证据必须由本策略、本宇宙的一次可复现运行机械产出。跨家族照抄证据、`nine_gate={}`、`gate=None`、低报 `n_trials` 都是作废级问题。

### 5.5 守卫和测试

关键守卫包括:

- `check_layer_deps.py`: 单向依赖和架构边界。
- `check_holdout_compliance.py`: holdout 截断、boundary 锁和迁移纪律。
- `check_registry_evidence.py`: 台账证据完整性和 DSR/9-Gate 约束。
- `check_lake_writers.py`: 数据湖写入口。
- `check_no_legacy_data.py`: 禁止正式路径依赖旧 `data_full`。
- `scripts/test_all.sh`: 顶层回归信号。

## 6. Web 产品面: 决策驾驶舱

前端的定位不是"展示数据的看板", 而是"每个页面回答一个决策"。

当前驾驶舱地图:

1. Alpha 工厂: 下一轮研究算力投向哪里? 哪个候选值得推进?
2. 验证闸门: 这个候选能否独立验证通过并入册?
3. PM 交易台: 今天能交易吗? 如果不能, 阻断证据是什么?
4. 数据与成本可信层: 数据、成本、时效和口径是否可信?
5. 配置层: 成本、模型、风控和 API 配置。

页面原则:

- 答案先行: 首屏先给裁决, 再给证据。
- 单一真相源: 每个红绿灯都能追到 registry、deployment、hash、data vintage 或 readiness。
- 研究不得自评: PASS 必须等同于台账和验证层裁决。
- 不用空 P&L 曲线充门面。
- 当前如果没有可部署策略, 页面应诚实显示 fail-closed, 而不是造假绿灯。

## 7. 当前真实状态

根据 `STATUS.md` 的当前记录, 系统已经具备:

- 全市场数据湖和 canonical 数据质量机制。
- 统一回测内核。
- AutoResearch、候选工厂、L0-L3 与 9-Gate 审计。
- holdout、trial ledger、DSR、PBO、衰减监控等治理设施。
- FastAPI + Next.js 产品面。
- 自动模拟盘和国债 ETF 防守/闲置资金管理。

但当前真实投资状态必须诚实表述:

- 近期多轮重审显示许多旧 standalone 策略不能通过 DSR 或衰减审计。
- 当前系统价值重心在研究工厂和验证闸门, 不是积极交易。
- fail-closed 是正确状态: 如果没有正式可部署 alpha, 系统应拒绝给出交易绿灯。
- 纸面前向和观察版本可以继续积累证据, 但不得包装成正式在册策略。

## 8. 系统边界

系统能做:

- 自动更新数据、生成信号和模拟盘。
- 批量生成候选并跑廉价验证。
- 对候选做统一回测、成本敏感性和门禁审计。
- 维护策略生命周期台账和证据链。
- 在 Web 中解释研究、风险、数据和部署状态。

系统不能或不应做:

- 自动真实下单。
- 用 LLM 判断 alpha 有效。
- 为达标修改真实成本、样本、shift、T+1 或股票池。
- 绕过 workflow 直接写台账。
- 用旧数据或临时脚本结果作为正式绩效。

## 9. 给非开发者的解释

可以把它理解为一家小型量化研究机构的操作系统:

- 数据部门: 负责把市场、财务、资金、ETF 数据整理成可信原料。
- 研究部门: 提出大量因子和策略想法。
- 验证部门: 用统一标准打假, 先假设它们都是错的。
- 投委会/台账: 只有证据完整、风险可解释的版本才能登记。
- 交易台: 在没有合格策略时宁可空仓/防守, 不为了让页面好看而交易。
- 审计部门: 记录每次试错、每次入册、每次退役, 防止事后改口。

这套系统的优势不是承诺永远赚钱, 而是把"别骗自己"变成工程约束。
