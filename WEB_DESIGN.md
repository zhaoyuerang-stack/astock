## 0.A 研究域信息架构覆盖规则（2026-06-21）

本节是当前实现的权威边界，覆盖本文后续“页面六：因子研究”和“页面七：研究实验”中与之冲突的旧描述。

研究生命周期固定为：

```text
研究素材 → 草案/候选 → L0-L3 → 人工复核 → 正式登记 → 因子审计与监控
```

页面职责：

| 页面 | 唯一职责 |
|---|---|
| `/experiments` | 登记前工作队列；按“待复核 → 阻塞 → 可执行 → 运行中”排序 |
| `/experiments/evidence` | 研报逻辑链与知识图谱；只能创建研究草案，不能把纯叙事直接送入回测 |
| `/experiments/runs` | 异步 Job、L0-L3 运行、失败原因和数据 vintage |
| `/experiments/reviews` | 人工批准/拒绝及正式晋级入口 |
| `/experiments/[kind]/[id]` | 单个草案、Hypothesis 或 AutoResearch 候选的完整档案 |
| `/factors` | 已进入策略台账的版本、家族血缘和 Nine-Gate 审计 |
| `/factors/[family]/[version]` | 版本完整档案，包括专项验证、影子表现、成本、容量和失效边界 |

硬边界：

1. 实验室不得重复展示已登记策略排行榜或第二套候选漏斗。
2. 因子页不得展示未登记 Hypothesis/AutoResearch 候选；台账中的 `候选/SHADOW/观察` 统一显示为“观察版本”。
3. 人工与 AutoResearch 候选通过 L3 后都必须留下人工批准记录，才能调用 `workflow.promote`。
4. Web 端不计算因子和绩效，不直接写 `strategy_versions.json`；所有晋级仍走台账唯一写入口。
5. 择时敏感度和影子表现按 `family/version` 关联到版本详情，不再作为实验室一级 Tab。

## 0. 文档目标

本文档用于定义“个人量化研究分析平台”的 Web 端产品设计、页面结构、组件体系、交互逻辑和前端实现边界。

平台不是传统行情看板，也不是单纯回测工具，而是一个面向个人量化研究的研究操作系统。Web 端的核心任务是把用户的投资研究过程显式组织为：

```text
假设发现 → 变量/因子构建 → 状态识别 → 策略构建 → 回测验证 → 执行监控 → 复盘迭代
```

并通过右侧常驻 AI Agent，将数据解释、实验审查、策略归因、风险预警和下一步建议嵌入用户工作流。

## 1. Web 端设计总原则

### 1.1 三栏布局优先

平台采用稳定三栏结构：

```text
┌──────────────┬──────────────────────────────┬──────────────────┐
│ 左侧导航栏   │ 中央研究工作区                 │ 右侧 AI Agent 栏  │
│ Navigation   │ Main Research Workspace       │ Agent Copilot    │
└──────────────┴──────────────────────────────┴──────────────────┘
```

三栏职责如下：

| 区域 | 职责 | 是否常驻 | 主要用户行为 |
|---|---|---|---|
| 左侧导航栏 | 模块切换、系统定位、状态提示 | 是 | 切换页面、查看系统状态 |
| 中央研究工作区 | 承载当前页面的核心分析、图表、表格、配置与执行 | 是 | 分析、筛选、回测、配置、查看结果 |
| 右侧 AI Agent 栏 | 上下文解释、研究建议、任务协同、快速操作 | 是，支持折叠 | 提问、生成报告、解释异常、启动任务 |

采用三栏布局的原因：

1. 量化研究任务复杂，用户需要同时查看数据、图表、结果和解释。
2. AI Agent 不应被隐藏在弹窗里，而应作为常驻协作对象。
3. 右侧 Agent 可以持续感知当前页面上下文，减少用户重复描述。
4. 左侧导航保持稳定，有助于非开发型用户建立空间记忆。
5. 中央工作区保留最大宽度，用于高密度金融图表和表格。

### 1.2 研究优先，不以交易为中心

Web 端默认围绕研究任务设计，而不是围绕下单交易设计。

优先级：

```text
数据可信 > 假设清晰 > 因子有效 > 策略稳健 > 风险可控 > 决策可解释 > 交易执行
```

因此页面设计中需要突出：

1. 研究流程。
2. 数据质量。
3. 实验记录。
4. 策略健康度。
5. 风险预算。
6. Agent 审查意见。

### 1.3 AI 是研究协作者，不是交易指令源

AI Agent 的定位：

```text
解释器：解释因子、回测、风险和组合变化。
审查员：发现未来函数、过拟合、样本外失效、数据异常。
研究员：生成假设、实验计划、研究报告和后续实验。
控制助手：提示风险超限、熔断条件和调仓建议。
```

AI Agent 不直接输出“买入/卖出”指令。所有涉及交易动作的内容必须表达为：

```text
研究建议
风险提示
调仓候选
执行前检查
```

### 1.4 高密度但不拥挤

量化平台天然信息密度高，设计目标不是极简，而是“可读的高密度”。

视觉原则：

1. 卡片化承载复杂信息。
2. 关键指标使用大字号。
3. 复杂表格支持筛选、排序和列配置。
4. 图表优先显示趋势，详细数值通过 hover 或详情页展示。
5. 风险类信息使用红/橙/黄分级，但避免全屏报警化。
6. 所有页面保持统一布局骨架，减少认知成本。

### 2.5 右侧 AI Agent 栏设计

右侧 Agent 是所有页面的固定协作层。

基础结构：

```text
Agent Header
当前任务 / 上下文
对话区
快捷操作
建议下一步
关注列表 / 预警
输入框
```

不同页面下，Agent 的能力域不同：

| 当前页面 | Agent 角色 |
|---|---|
| 总览 | 总览分析助手、风险解释、日报生成 |
| 数据中心 | 数据 QA 助手、异常解释、修复建议 |
| 因子研究 | 因子分析助手、IC 解读、因子改进建议 |
| 策略回测 | 回测审查助手、过拟合检查、策略诊断 |
| 组合管理 | 组合优化助手、调仓解释、what-if 分析 |
| 风险控制 | 风控助手、风险来源解释、控制措施建议 |
| 研究实验 | 实验助手、实验计划生成、结果总结 |
| AI研究助手 | 主工作台，Agent 中央化 |
| 系统设置 | 配置助手、最佳实践建议、安全检查 |

Agent 交互原则：

1. Agent 必须知道当前页面、当前选中对象和当前用户任务。
2. Agent 输出需要结构化，优先使用卡片、列表、按钮和引用对象。
3. 所有 Agent 建议都需要区分“解释”“建议”“可执行操作”。
4. 涉及回测、组合、风控的建议必须附带原因和风险说明。
5. 高风险动作需要二次确认，不允许一键执行实盘动作。

## 14. 关键交互设计

### 14.1 当前页面上下文同步给 Agent

当用户切换页面或选中对象时，系统自动更新 Agent 上下文。

上下文对象：

```json
{
  "current_page": "factor_research",
  "selected_object_type": "factor",
  "selected_object_id": "momentum_12m",
  "date_range": "2023-01-01~2025-05-23",
  "universe": "CSI_ALL",
  "active_filters": {},
  "recent_actions": []
}
```

### 14.2 从 Agent 发起操作

Agent 的操作分级：

| 操作级别 | 示例 | 是否需要确认 |
|---|---|---|
| 只读 | 解释 IC、生成摘要 | 否 |
| 低风险 | 生成报告、创建实验草稿 | 否或轻确认 |
| 中风险 | 修改回测参数、启动批量实验 | 是 |
| 高风险 | 调仓、生成订单、修改风险规则 | 必须二次确认 |

### 14.3 页面对象引用

用户可以在 Agent 中引用对象：

```text
@动量因子
@多因子增强策略
@当前组合
@风险预警#123
@实验#E20250523001
```

### 14.4 报告生成

报告生成入口：

```text
总览页：生成每日研究报告
因子页：生成因子研究报告
回测页：生成策略回测报告
风控页：生成风控报告
实验页：生成实验总结报告
```

报告状态：

```text
待生成
生成中
已完成
失败
已归档
```

### 14.5 异常状态

数据异常：

```text
数据缺失
接口失败
数据延迟
复权异常
停牌识别失败
```

展示方式：

1. 顶部通知。
2. 数据中心红/橙色状态。
3. 相关研究页面提示“数据质量不足，不建议运行回测”。
4. Agent 自动生成异常解释。

### 15.4 API 设计原则

前端不直接处理核心计算逻辑，所有量化计算由后端服务提供。

前端职责：

```text
展示
筛选
配置
提交任务
查看状态
触发报告
与 Agent 交互
```

后端职责：

```text
数据计算
回测执行
因子计算
风险计算
报告生成
Agent 工具调用
权限与审计
```

## 17. 验收标准

### 17.1 布局验收

1. 左侧导航、中央工作区、右侧 Agent 三栏结构稳定。
2. 右侧 Agent 在页面切换后保持存在。
3. 中央区域可承载大表格和复杂图表。
4. 1280px 宽度下不出现严重遮挡。
5. 主要页面视觉风格统一。

### 17.2 信息架构验收

1. 用户能在 3 秒内识别当前页面属于哪个模块。
2. 用户能从总览页进入数据、因子、回测、风险、组合页面。
3. 每个页面至少包含一个清晰主任务。
4. Agent 能显示当前页面上下文。

### 17.3 交互验收

1. 筛选器改变后，图表和表格状态同步更新。
2. 表格支持排序、筛选、分页。
3. 图表支持 hover 查看详情。
4. Agent 快捷操作能打开对应任务或生成报告。
5. 高风险操作必须二次确认。

### 17.4 视觉验收

1. 指标卡片、图表、表格、Agent 卡片风格一致。
2. 风险状态颜色清晰但不过度刺眼。
3. 数字、单位、时间、状态标签格式统一。
4. 金融指标对齐清晰，表格可读。

## 19. 最终设计判断

Web 端的核心设计不是“展示更多行情数据”，而是构建一个持续运行的个人研究控制台。

最终界面要稳定回答以下问题：

```text
我现在研究什么？
依据的数据是否可信？
因子是否有效？
策略是否稳健？
组合是否失控？
AI 给出的解释和下一步建议是什么？
这次研究如何沉淀为可复用资产？
```

因此，本平台 Web 端应坚持：

```text
三栏结构
研究流程显性化
右侧 Agent 常驻
数据质量前置
风险控制内嵌
实验结论可沉淀
```

这套设计的最终目标是让用户不是“看盘”，而是在一个受控、可复现、可解释的环境中完成个人量化研究。
